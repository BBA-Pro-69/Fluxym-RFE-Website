Flux 1 — Données réglementaires de facture

PA émettrice → PPF UBL / CII (XML)

À chaque facture émise, la PA-E extrait les données réglementaires et les transmet au PPF : identifiants vendeur/acheteur (SIREN, TVA intracom), numéro et date de facture, montants HT/TVA/TTC par taux, type de document, cadre de facturation.

📌
Trajectoire : profil « Base » en sept. 2026 (données réduites) → profil « Full » en sept. 2027 (données complètes).

Flux 2 — Transmission de la facture

PA émettrice → PA réceptrice UBL / CII / Factur-X

La facture complète circule directement entre PA. Ce flux ne transite PAS par le PPF. Transport via convention bilatérale entre PA ou réseau Peppol.

Flux 3 — Mise à disposition au destinataire

PA réceptrice → Destinataire Format au choix du destinataire

Le « dernier kilomètre » : la PA-R met la facture à disposition du destinataire. Conversion de format possible (portail web, API, EDI, email sécurisé…).

⚠️ Flux 4 et Flux 5 — INEXISTANTS

⚠️
Les Flux 4 et 5 n'existent pas.

Dans le modèle initial, quand le PPF devait jouer le rôle de routeur central :
• F4 = transmission PA-E → PPF (pour routage)
• F5 = retransmission PPF → PA-R

Avec l'abandon du PPF-routeur (oct. 2024), ces flux sont caducs. La numérotation a été conservée avec ces « trous » pour ne pas casser les références existantes.

Flux 6 — Statuts de cycle de vie

PA → PPF UN/CEFACT CDAR

À chaque changement de statut obligatoire, la PA transmet au PPF :

212EncaisséeExigibilité TVA à l'encaissement
213RejetéeFacture n'entre pas dans le circuit

Flux 7, 8, 9 — Annuaire

F7 : PA → PPF API REST

Consultation ponctuelle — vérifier l'existence du destinataire, sa PA-R, son code routage avant chaque émission.

F8 : PA → PPF API REST

Mise à jour annuaire — la PA-R crée, modifie ou clôture les lignes d'annuaire de ses clients en réception.

F9 : PA → PPF API REST

Consultation étendue — vérifier ses propres données ou celles de partenaires dans l'annuaire.

Flux 11 — Retours de contrôle PPF

PPF → PA API / EDI

Le « feedback loop » : pour chaque Flux 1, 6 ou 10, le PPF renvoie un accusé (OK) ou un rejet (KO).

5 motifs de rejet :

  • REJ_INC — Incohérence dans les données
  • REJ_INEX — Non-conformité (données inexistantes)
  • REJ_RG — Règle de gestion violée
  • REJ_HAB — Défaut d'habilitation
  • REJ_ENCAISSEMENT — Montants d'encaissement non conformes

Flux 12 — Transmission à la DGFiP

PPF → DGFiP Interne administration

Toutes les données fiscales agrégées (F1 + F6 + F10) sont transmises à l'administration fiscale. Alimente le pré-remplissage TVA. Invisible pour les entreprises.

Flux 13 & 14 — Gestion technique de l'annuaire

F13 : PA → PPF API / EDI

Actualisation en masse — quand une PA prend en charge de nouveaux clients ou qu'un client change de PA.

F14 : PPF → PA XML F14

Diffusion de l'annuaire — complet (hebdomadaire) + différentiel (quotidien). Permet aux PA de synchroniser leur copie locale.
